    Read index.html in full first to locate the exact insertion points before making changes.

    Make three additions to index.html:

    1. HIDDEN FIELD — inside the results block (where show_results is truthy), after the
       reset banner and before the lineup tables, add a standalone hidden input that carries
       card_pool across page renders. This is a belt-and-suspenders approach — the Re-Optimize
       form also carries card_pool, but this input ensures the value is available to any future
       form in the results section:

       ```html
       {% if show_results and card_pool_json %}
       <input type="hidden" id="card-pool-data" name="card_pool" value="{{ card_pool_json | e }}" />
       {% endif %}
       ```

       Use | e (not | safe) to HTML-entity-escape the JSON value for the attribute context.
       Do NOT use | tojson (double-encoding). Do NOT use | safe (disables escaping of user data).

    2. RE-OPTIMIZE FORM — immediately before the lineup results, OUTSIDE the upload
       <details id="upload-section"> block, guarded by both show_results and card_pool_json:

       ```html
       {% if show_results and card_pool_json %}
       <form method="post" action="{{ url_for('main.reoptimize') }}" id="reoptimize-form">
         <input type="hidden" name="card_pool" value="{{ card_pool_json | e }}" />
         <button type="submit">Re-Optimize</button>
       </form>
       {% endif %}
       ```

       Placement rule (from CONTEXT.md, locked decision): above lineup results, first thing
       user sees after results load, NOT inside the <details> upload section.

    3. JS LISTENER — in the existing script block at the bottom of the page, add a guarded
       listener for the re-optimize form after the existing form listener:

       ```javascript
       const rf = document.getElementById("reoptimize-form");
       if (rf) {
         rf.addEventListener("submit", function () {
           document.getElementById("loading-overlay").style.display = "flex";
         });
       }
       ```

       The null guard (if rf) is required because the form element is conditional — on GET
       and error pages it does not exist in the DOM. Without the guard, addEventListener
       throws TypeError on page load when results are not shown.

       No new CSS needed. No new JS files. Reuse existing overlay element as-is.
